# Generated CDP code policy

The entire `zendriver/cdp/` package is auto-generated. Do not hand-edit any file inside it.

## How it was generated

The modules were generated by [PyCDP](https://github.com/hyperiongray/python-cdp) from Chrome DevTools Protocol JSON schema definitions. The generator produces typed Python modules for each CDP domain, including:
- command methods (return generators)
- event types
- result types
- parameter types

## What not to do

- Do not add custom methods to generated files
- Do not fix type annotations by hand
- Do not rename generated classes
- Do not add `__init__.py` imports manually

## What to do instead

If a CDP type or command is missing, outdated, or wrong:

1. Check whether the issue is in the upstream CDP schema or in the generator
2. Fix upstream or the generator, not the generated output
3. Run the regeneration script (see `scripts/generate_cdp.py` or equivalent in this repo)
4. Commit the regenerated result

If you need helper behavior that the generated code does not provide, add it in `zendriver/core/` (e.g., `tab.py`, `element.py`, `util.py`) rather than extending generated modules.

## Why this matters

Generated code is large (~80+ files) and tightly coupled to Chrome's protocol version. Hand-edits create drift that is silently overwritten later, leading to:
- lost bug fixes
- merge conflicts
- misleading type signatures
- confusion about which behavior is "real"

The README in `zendriver/cdp/README.md` explicitly states: "do not modify their contents because the changes will be overwritten in next generations." Respect this.
