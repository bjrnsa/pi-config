# Zendriver anti-patterns and common mistakes

Use this reference when users are stuck, tests are behaving weirdly, or stealth is not working. These are the most frequent sources of confusion in the Zendriver codebase.

## Do not hand-edit `zendriver/cdp/`

The entire `zendriver/cdp/` package is auto-generated by PyCDP from Chrome DevTools Protocol schema definitions. Any hand-edited changes will be overwritten on the next regeneration.

If a CDP type is missing or wrong, the correct fix is usually:
1. Update the CDP schema or PyCDP generator
2. Run `scripts/generate_cdp.py` (or equivalent regeneration script)
3. Commit the regenerated output

Never add custom methods to generated CDP modules.

## Expert mode is not stealth mode

`Config(expert=True)` adds:
- `--disable-web-security`
- `--disable-site-isolation-trials`
- a script that forces all future `attachShadow()` calls to use `mode: "open"`

These changes make the browser **more detectable**, not less. A site that checks for normal security headers or closed shadow roots will see an abnormal browser.

Expert mode exists for:
- debugging shadow-DOM content
- bypassing same-origin restrictions during local development
- inspecting otherwise inaccessible DOM trees

It should **never** be recommended for anti-bot scraping.

## Windows event loop policy

On Windows, Python's default `ProactorEventLoop` is incompatible with some `asyncio` patterns used by `websockets` and the listener loop. This causes hangs or `RuntimeError` during browser startup.

Required at the top of any Windows script:

```python
import asyncio
import sys

if sys.platform == "win32":
    asyncio.set_event_loop_policy(asyncio.WindowsSelectorEventLoopPolicy())
```

The test suite (`tests/conftest.py`) sets this automatically in fixtures. Production scripts must set it manually.

## React and modern framework inputs

`Element.clear_input()` and `Element.clear_input_by_deleting()` exist specifically because React and similar frameworks use `_valueTracker` to intercept direct `element.value = ""` assignments. When the tracker sees the same value it already knows, it skips firing `onChange`.

Zendriver's solution:
- `clear_input()` uses `Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value').set.call(el, '')` to bypass the instance-level tracker, then dispatches an `InputEvent`
- `clear_input_by_deleting()` simulates backspace keypresses with the same native setter approach

Anti-pattern: writing custom JS that does `element.value = ""` without understanding why it fails on React.

## Deprecated APIs

| Old | New | Version |
|-----|-----|---------|
| `Element.__getattr__(name)` | `Element.get(name)` | v0.13.0 |
| `Connection.__getattr__` | declared `@property` accessors | v0.13.0 |
| `StrEnum` usage | plain `Enum` / `str, Enum` | v0.12.1 |
| `util.loop()` | `asyncio.run()` directly | v0.5.1 |

Code changes in the repo should avoid deprecated patterns. When reviewing user code, suggest migration.

## Headless UA patching is partial

When running headless, Zendriver patches `navigator.userAgent` to remove the word `Headless`. This fools naive checks but does not touch:
- `navigator.plugins` length and mime types
- `navigator.webdriver` (absent in CDP, which is good, but other fingerprint vectors exist)
- WebGL renderer strings
- screen/window dimension consistency
- permission prompt timing
- Chrome DevTools protocol presence itself

Do not tell users that headless mode is "undetectable." It is "less detectable than WebDriver headless."

## `await tab` vs `await tab.sleep()`

`await tab` waits for the CDP listener to go idle (no new protocol messages). This is usually correct after navigation or DOM mutations.

`await tab.sleep(t)` unconditionally sleeps plus refreshes target info. Use this when you know the site needs a fixed delay (e.g., animation, lazy loading timer).

Anti-pattern: replacing all `await tab` calls with `await asyncio.sleep(5)` because "the page is slow." This makes scripts slow and flaky.

## `find()` matches script contents

`tab.find("login")` searches all text nodes, including inline `<script>` tags and `<meta>` descriptions. This often returns noise.

Prefer:
- `tab.find("login", best_match=True)` — uses text-length heuristic to pick the closest match
- `tab.select("button[type=submit]")` — CSS selector when markup is stable
- `tab.xpath(...)` — when you need precise DOM traversal

Anti-pattern: using `find()` without `best_match` on pages with heavy inline scripts.

## `intercept()` blocks until resumed

When `intercept()` matches a request, the browser pauses that request. If your script registers the interceptor but never calls `continue_request()` or `fulfill_request()`, the page will hang.

Always use `intercept()` as a context manager or ensure you explicitly resume:

```python
async with tab.intercept("*", RequestStage.REQUEST, ResourceType.XHR) as interceptor:
    event = await interceptor.response_future
    await interceptor.continue_request()
```

Anti-pattern: setting up an interceptor, awaiting a future, and forgetting to resume.

## `expect_*` patterns need regex full-match

`expect_request("api/v1")` will **not** match `https://example.com/api/v1/users`. The pattern is compiled with `re.fullmatch()`, so it must match the entire URL.

Correct: `expect_request(r".*api/v1.*")`

## `Browser.__aenter__` is not `browser.stop()`

`Browser.__aenter__` and `__aexit__` are minimal implementations. `__aexit__` does not call `stop()`. Relying on `async with browser:` for cleanup is unsafe.

Always explicitly call `await browser.stop()` at the end of scripts.

## Temp profile cleanup races

Chrome parent processes can survive `Browser.stop()` if they were already dying or if a child process holds the profile directory open. `_cleanup_temporary_profile()` retries 5 times with 0.15s delays, but may still log a warning.

If you see profile cleanup errors, check for zombie Chrome processes.

## Cookie save/load pattern filtering is subtle

`browser.cookies.save(file, pattern=".*")` filters cookies by regex matching against the **stringified dict** of each cookie. This means the pattern matches domain, name, and value fields simultaneously.

Debugging a cookie that "should have been saved" often fails because the pattern did not match the stringified representation.

Prefer `pattern=".*"` (all cookies) when debugging, then narrow down once you confirm the save path works.
